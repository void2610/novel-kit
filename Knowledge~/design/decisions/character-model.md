# 状況

キャラ表示モデルは永続的な API 境界になる。ジャンル標準（Naninovel/Utage）は
base + 表情 + ポーズ等のレイヤ合成と left/center/right 等のステージ配置を持つが、
調査した全プロジェクトは単一スプライト差し替え（`set_portrait`）にとどまる。

# 決定

**単一スプライト差し替え**を採る。

- `portrait(char, name)` で名前付き 1 枚スプライトを単一スロットに差し替える。
- 多層合成（base+表情+ポーズ+口）・複数キャラ同時配置・move/focus は v1 では持たない。

```
portrait(char, "笑顔_制服")  # 1 枚差し替え
表情変更も全体指定 / 単一スロット
```

# 理由

- 実装が軽量。既存実装の方式を踏襲でき、調査済みの void-red の 2 画像クロスフェード View を
  そのまま単一スロット描画に流用できる。
- ユーザー判断（多層合成/ステージ配置の表現力より軽量さを優先）。

# 帰結

- 「表情だけ変更」は全体スプライト指定で行う（差分表情の独立変更は不可）。
- 複数キャラの同時配置・位置移動は v1 のコマンド/`IPortraitView` 契約に含めない。
- 表現力上限を将来引き上げる場合、コマンドと View 契約の拡張（後方互換に注意）が必要。
- 描画は void-red の 2 画像 true クロスフェード（スプライト pop 回避）+ slide-in を採用候補とする。

# 検討した代替案

- **多層合成 + ステージ配置**（当初推奨）: Naninovel/Utage 相当で表現力上限を確保するが実装重。不採用。
- **単一コア + View は多層を見据え設計**: DSL は単一だが `IPortraitView` を多層対応で設計。
  v1 を軽くしつつ将来余地を残す折衷案だったが、単一スプライト方式に確定。
