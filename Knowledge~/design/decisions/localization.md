# 状況

調査した全プロジェクトでシナリオテキストは `.rb` 内の Ruby 文字列リテラルにハードコードされている。
ロケール切替を一級にするにはテキストのキー外部化が要るが、それは「著者が Ruby でプロセを直書きする」
DSL と構造的に衝突する。これは DSL 全体を規定する v1 判断である。

# 決定

**日本語のみ + 抽出フック**を採る。

- `.rb` にプロセ直書きを維持する（著者体験を変えない）。
- `say` のテキストを `ITextResolver` で差し替えられるフックを**最初から設計**しておく。
- 多言語対応は将来、このフック経由で**非破壊に後付け**する。

```
say "本気なの？"        # .rb に直書き維持
+ ITextResolver フック  # key 化は将来差し替え可
v1 は日本語のみ / 多言語は非破壊後付け
```

# 理由

- 著者体験（プロセ直書き）を損なわずに、将来の多言語化の seam を安価に確保できる。
- v1 のスコープを日本語に絞り、実装と設計判断を軽くできる。

# 帰結

- `say` ハンドラはテキストを直接使わず `ITextResolver` を通す（既定は恒等変換）。
- 将来のキー外部化やロケール別解決は `ITextResolver` 実装の差し替えで実現する。
- ルビ（ふりがな）は JP テキストのスタイリングであり、ローカライズ機構とは別扱い（任意モジュール）。

# 検討した代替案

- **キー外部化を v1 から**: テキストをキー化しロケール表から解決。多言語が一級だが著者がプロセ直書き
  できなくなる。不採用。
- **ロケール別 `.rb` セット**: 言語ごとにシナリオファイルを分け `IScenarioSource` がロケールで選択。
  プロセ直書きは保てるが翻訳同期が手動。将来 `ITextResolver` 方式と併用可能だが v1 では採らない。
