# 状況

[実行モデル](/design/decisions/execution-model.md) でリプレイ前提設計を採った。リプレイ再実行で
同じ地点に着くには本来シナリオの**決定性**が要る（`rand`/`Time.now` の直接使用は再現性を壊す）。
ただし当面の復帰はチェックポイント単位であり、完全な save-anywhere はまだ実装しない。

# 決定

**決定性コントラクトの強制は当面後回し**にする。

- 選択/フラグの**入力履歴は day 1 から記録**する（実行モデルの決定どおり）。
- ただし「`rand`/`Time` 直接使用の禁止」「seed 付き RNG / 論理時刻 API の提供」といった
  著者への決定性強制は**現時点では課さない**。
- 厳格なリプレイ契約は、将来 save-anywhere を実装する段階で規定する。

```
当面: チェックポイント復帰のみ
決定性契約は save-anywhere 実装時に後回し
```

# 理由

- 当面の復帰範囲ではフルな決定性を要求する必要がなく、著者体験を縛らずに進められる。
- 履歴記録という安価な土台だけ先に確保しておけば、契約の後付けは可能。

# 帰結（リスク）

- 後から著者に決定性制約を課す場合、**既存 `.rb` シナリオの修正が必要**になるリスクがある。
- save-anywhere 実装時に、当時のシナリオが非決定的だと完全再現できないケースが出得る。
- 緩和策として、将来の制約導入を見越し、決定性 API（`novel_rand` 等）を早期に「推奨」として
  提示しておくことを検討する（強制はしない）。

# 検討した代替案

- **厳格 + seed 付き API 提供**（当初推奨）: 完全再現を保証するが著者を縛る。当面の範囲では過剰として見送り。
- **実用: 状態問合せも記録**: 非決定性を許容しつつ戻り値も履歴化。セーブデータ肥大化のため見送り。
